# 在“辅助功能”设置页追加 BetterActionBar 按钮的实现方案

## 目标

- 将 BetterActionBar 的配置入口放入游戏 `辅助功能` 页面。
- 入口以 `按钮控件` 形式出现，并且位于原版选项列表之后（尾部追加）。
- 保持当前项目的多版本结构（`src/versioned/*`）可维护。

## 项目现状与约束

- 当前模组配置由 `src/client/java/io/github/mooling0602/betteractionbar/client/BetterActionBarConfig.java` 负责读写，已支持：
  - `LineSpacingMultiplier`
  - `NewLineBreak`
- 目前没有设置界面；`TODO.md` 第 6 项已明确要在 `辅助功能` 中修改配置。
- 项目已采用“同名 Mixin + 版本目录覆盖”的方式处理版本差异（如 `ActionBarNewlineMixin`）。

## 推荐方案（主方案）

### 1) 注入点选择：`AccessibilityOptionsScreen#addOptions` 尾部

核心思想：在原版把所有辅助功能选项加入列表后，再追加一行自定义控件，天然满足“追加到原版选项后”。

- 目标类：`AccessibilityOptionsScreen`
- 目标方法：`addOptions`（或该版本对应映射名）
- 注入时机：`@At("TAIL")`

这样做的优点：

- 顺序稳定：不会插到原版中间。
- 侵入性小：不改原版选项构造流程。
- 语义清晰：就是“最后再加一个按钮”。

### 2) 访问选项列表容器：通过 `GameOptionsScreen` 的 Accessor

从 Yarn 映射可见，`GameOptionsScreen` 内有 `OptionListWidget body`（列表容器）。
在 Mixin 里推荐用 Accessor 暴露该字段，再调用列表 API 追加控件。

建议新增：

- `GameOptionsScreenAccessor`（`@Mixin(GameOptionsScreen.class)`）
- `@Accessor("body") OptionListWidget betterActionBar$getBody();`

### 3) 追加按钮的方式

`OptionListWidget` 提供了 `addWidgetEntry` / `addSingleOptionEntry` 等能力。对于“纯按钮入口”，推荐：

- 使用 `ButtonWidget`（或该版本对应按钮类）构建按钮；
- 将按钮通过 `addWidgetEntry(button, EmptyWidget)` 或同等双列 API 放入列表；
- 按钮点击后打开 `BetterActionBar` 自定义子页面（或先做最小可用：弹出提示并打开子页）。

> 若某版本没有“空右列”便捷控件，可创建宽度对齐的占位控件，保证 UI 行布局不塌。

## 代码结构建议（按本仓库风格）

### 目录与类

- 公共逻辑（不依赖具体 MC 内部签名）：
  - `src/client/java/.../client/gui/BetterActionBarAccessibilityScreen.java`
  - `src/client/java/.../client/gui/BetterActionBarAccessibilityUiBridge.java`
- 版本相关 Mixin（同名，分目录实现）：
  - `src/versioned/V_26_1/client/java/.../mixin/client/AccessibilityOptionsButtonMixin.java`
  - `src/versioned/V_26_2/client/java/.../mixin/client/AccessibilityOptionsButtonMixin.java`
  - `src/versioned/V_1_21_1/client/java/.../mixin/client/AccessibilityOptionsButtonMixin.java`
  - `src/versioned/V_1_20_6/client/java/.../mixin/client/AccessibilityOptionsButtonMixin.java`

并在 `src/client/resources/betteractionbar.client.mixins.json` 的 `client` 数组中追加：

- `AccessibilityOptionsButtonMixin`
- `GameOptionsScreenAccessor`（若 Accessor 放在公共 `src/client`）

## 关键伪代码（概念示例）

```java
@Mixin(AccessibilityOptionsScreen.class)
abstract class AccessibilityOptionsButtonMixin {

    @Inject(method = "addOptions", at = @At("TAIL"))
    private void betterActionBar$appendButton(CallbackInfo ci) {
        OptionListWidget list = ((GameOptionsScreenAccessor) this).betterActionBar$getBody();
        if (list == null) return;

        MinecraftClient client = MinecraftClient.getInstance();
        ButtonWidget button = ButtonWidget.builder(
                Text.translatable("betteractionbar.options.open"),
                b -> client.setScreen(new BetterActionBarAccessibilityScreen((Screen) (Object) this))
            )
            .width(150)
            .build();

        list.addWidgetEntry(button, EmptyWidget.ofWidth(150));
    }
}
```

说明：

- 真实类名/方法名需按 `mcTarget` 对应映射调整。
- 某些版本按钮构造器不是 `builder(...)`，需要改为该版本 API。
- 若方法名不是 `addOptions`，保持“原版选项加入完成后再注入”的原则即可。

## 与现有配置系统的衔接

按钮进入子页后可直接复用 `BetterActionBarConfig`：

1. 页面初始化读取 `BetterActionBarConfig.get()`；
2. 控件变更时更新内存值；
3. 点击“完成/应用”时写回配置文件（可在 `BetterActionBarConfig` 增加显式 `save(...)`）；
4. 返回父页面时触发 `BetterActionBarConfig.reload()`（或已写内存则无需 reload）。

建议第一阶段只做两项控件：

- 行距（`LineSpacingMultiplier`，滑块）；
- 自定义换行符（`NewLineBreak`，按钮进入二级编辑页或文本输入页）。

## 版本差异处理建议

本仓库已同时覆盖 `26.2 / 26.1 / 1.21.1 / 1.20.6`，建议沿用现有策略：

- Mixin 类名保持一致，放到 `src/versioned/<variant>/...`；
- 每个版本只处理“方法签名、控件构造器、列表追加 API”差异；
- 业务逻辑尽量下沉到 `src/client` 公共类，降低重复代码。

## 风险点与规避

- 映射/签名变化导致注入失败：
  - 规避：每版本单独 Mixin，`require=1` 保持严格并在开发期尽早暴露问题。
- 按钮布局在不同分辨率错位：
  - 规避：优先走 `OptionListWidget` 原生行布局，不手动绝对定位。
- 配置写回时机不一致：
  - 规避：统一“完成按钮触发保存”，页面关闭不自动写盘。

## 最小落地路径（建议按此实现）

1. 先实现“辅助功能页尾部出现 BetterActionBar 按钮，点击可打开空白子页”。
2. 再在子页接入 `LineSpacingMultiplier` 滑块并保存。
3. 最后补 `NewLineBreak` 编辑能力。

该路径能最快闭环 `TODO.md` 的第 6 项，同时把复杂度拆分到可验证的小步骤。
